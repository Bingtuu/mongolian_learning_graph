# CLAUDE.md — 蒙古语学习知识库编码规范与工作流

> **作用域**: 本文件对项目根目录及所有子目录生效。
> **目标读者**: AI 编码助手（Claude / Kimi 等）。
> **核心理念**: 知识库是**单一事实来源（Single Source of Truth）**，所有规则确保一致性、可追溯性和可维护性。

---

## 1. 项目概述

- **目标**: 面向中文学习者的蒙古语（西里尔）结构化知识库
- **技术栈**: Obsidian Markdown + YAML Frontmatter + 双向链接
- **工作语言**: 中文为主，蒙古语（西里尔）为辅，必要时标注拉丁转写
- **版本控制**: Git，主分支 `main`

### 2.3 Wikilink 规范

```markdown
<!-- 正确 -->
[[过去时]]
[[阶段1-字母与问候|阶段1：字母与问候练习]]  <!-- 显示别名 -->

<!-- 错误 -->
[[过去时.md]]       <!-- 不要加扩展名 -->
[[过去时 ]]         <!-- 不要末尾空格 -->
[[过去时|]]         <!-- 不要空别名 -->
```

- **链接目标** = 文件名（不含 `.md`）
- **显示别名** = 用 `|` 分隔，仅在需要不同显示文本时使用
- **区分大小写**: 在 wikilink 中保持与文件名一致的大小写

## 3. YAML Frontmatter 严格规范

每个 wiki 页面**必须**包含完整 frontmatter，缺失任何必需字段视为格式错误。

```yaml
---
title: 中文标题                    # 必需。页面的中文标题，与文件名一致
type: concept|entity|rule|comparison|source|example|synthesis|exercise|index|log
category: 分类                     # 必需。如"动词语法""名词语法""句法"
level: fundamental|beginner|intermediate|advanced  # 必需。学习难度
related: [[页面A]], [[页面B]]      # 可选但强烈建议。双向链接到相关页面
prerequisites: [[页面X]]           # 可选。学习本页面前需要了解的内容
source: [[源材料页面]]              # concept/entity/rule/comparison/synthesis/example/exercise 必需
status: draft|needs-check|needs-ocr|reviewed  # 必需（除 source/index/log 外）
---
```

### 3.1 字段详解

| 字段 | 类型 | 必需页面 | 说明 |
|------|------|----------|------|
| `title` | string | 全部 | 中文标题，与文件名一致 |
| `type` | enum | 全部 | 见下方"页面类型矩阵" |
| `category` | string | 全部 | 分类标签，用于索引分组 |
| `level` | enum | 全部 | `fundamental`（基础概念）、`beginner`、`intermediate`、`advanced` |
| `related` | list | 建议 | 相关页面双向链接 |
| `prerequisites` | list | 建议 | 前置知识 |
| `source` | list/string | 知识页 | 本页内容的来源。多来源用列表 |
| `status` | enum | 知识页 | `draft`（草稿）、`needs-check`（需人工校对）、`needs-ocr`（需 OCR）、`reviewed`（已审阅） |

### 3.2 页面类型矩阵

| 类型 | 用途 | 必需字段 | 典型内容 |
|------|------|----------|----------|
| `concept` | 抽象语言学概念 | title, type, category, level, status | 定义、理论、分析框架 |
| `entity` | 具体语言元素 | title, type, category, level, status | 列表、对照表、分类 |
| `rule` | 语法规则 | title, type, category, level, source, status | 规则概述、详情表格、例句、常见错误 |
| `comparison` | 易混淆知识点 | title, type, category, level, source, status | 使用场景、形式对比、最小对比例句 |
| `source` | 源材料摘要 | title, type, category, level | 基本信息、内容结构、已提取知识点、待提取清单 |
| `example` | 按主题例句集 | title, type, category, level, source, status | 蒙古语原文、发音、词素拆解、翻译、来源标注 |
| `synthesis` | 合成速查 | title, type, category, level, source, status | 覆盖范围、决策树、适用阶段 |
| `exercise` | 练习题 | title, type, category, level, source, status | 题目、答案、解析、关联知识点 |
| `index` | 导航入口 | title, type | 链接、摘要、学习路径 |
| `log` | 操作日志 | title, type | 时间戳、操作类型、变更列表 |

## 4. 内容格式规范

### 4.1 例句格式（强制标准）

所有例句**必须**遵循以下格式之一：

**标准格式**（最常用）：
```markdown
> Би эмч биш, багш.
> (我不是医生，是老师。)
> — [[日常问候]] | 语法点: [[否定式]]
```

**完整格式**（核心/疑难例句）：
```markdown
> Би энэ номыг уншсан.
> Bi ene nomyg unshsan.
> би / энэ / ном-ыг / унш-сан
> 我 / 这 / 书-[宾格] / 读-[过去时]
> (我读过这本书。)
> — 来源: [[Gaunt 现代蒙古语]] Ch.3 p.45 | 语法点: [[过去时]]、[[宾格]]
```

**字段顺序**：
1. 西里尔蒙古文（`>` 引用块）
2. 拉丁转写（可选，核心例句建议）
3. 词素拆解（可选，教学重点例句建议）
4. 中文翻译（括号包裹）
5. 来源与语法点标注（`— 来源: ... | 语法点: ...`）

### 4.2 术语标注规范

所有语法术语**必须**同时标注中文和蒙古语原文：

```markdown
[[元音和谐律]] (Vowel Harmony / эгшгийн зохицол)
[[过去时]] (Past Tense / өнгөрсөн цаг)
```

### 4.3 蒙古文规范

- **默认使用新蒙文（西里尔）**
- **老蒙文**仅在必要时使用，格式：
  ```markdown
  新蒙文: ном (书)
  老蒙文: {{ᠨᠣᠮ}}
  ```
- **拉丁转写**使用系统转写（非 IPA），如 `Mongol hel`

### 4.4 表格规范

- 表格必须有表头
- 表格内容中对齐
- 复杂表格添加说明文字

```markdown
| 不定式 | 过去时 | 含义 |
|--------|--------|------|
| авах | ав**сан** | 拿了 |
| олох | оло**сон** | 找到了 |
```

## 5. 源材料处理流程（核心规则）

这是知识库扩展的**第一道闸门**。所有新知识必须经过此流程。

### 5.1 源材料分级

| 级别 | 定义 | 处理优先级 |
|------|------|-----------|
| **Primary** | 权威教材、语法专著（如清格尔泰语法、Gaunt 教材） | P0 — 优先收录 |
| **Secondary** | 辅助教材、在线课程、词典 | P1 — 补充收录 |
| **Tertiary** | 博客、论坛、个人笔记 | P2 — 谨慎收录，需交叉验证 |

### 5.2 源材料收录检查清单（Ingest Checklist）

在从任何源材料提取知识前，**必须**完成以下检查：

```
□ 1. 源材料是否已在 sources/ 中有对应页面？
   ├─ 否 → 创建源摘要页（见 5.3）
   └─ 是 → 更新源摘要页的"已提取知识点"

□ 2. 该源材料的 OCR/文本提取质量是否可接受？
   ├─ 否 → 标记 status: needs-ocr，暂停收录
   └─ 是 → 继续

□ 3. 提取的知识点是否已有对应 wiki 页面？
   ├─ 已有页面 → 进入【合并流程】（见第 6 章）
   └─ 无页面 → 进入【新建流程】（见第 7 章）

□ 4. 是否更新 wiki/log.md？（见第 9 章）

□ 5. 是否更新 wiki/index.md？
```

### 5.3 源摘要页模板

每个源材料必须在 `wiki/sources/` 中有一个对应页面：

```yaml
---
title: 源材料名称
type: source
category: 教材|专著|在线资源|词典
level: beginner|intermediate|advanced
related: [[其他相关源材料]]
---

# 源材料名称

## 基本信息

- **作者**: 
- **语言**: 西里尔蒙文 / 老蒙文 / 中文 / 英文
- **来源**: URL 或文件路径
- **级别**: Primary / Secondary / Tertiary

> **说明**: 本页引用的并非...（如有转录/改编说明）

## 内容结构

### 第一章/Unit 1
- 知识点列表

### 第二章/Unit 2
- 知识点列表

## 已提取知识点

- [x] [[知识点A]] — 简要说明
- [ ] [[知识点B]] — 待提取

## 待提取内容

| 内容 | 优先级 | 阻塞原因 | 计划日期 |
|------|--------|----------|----------|
| Unit 3 食物 | P1 | 无 | 待定 |
| 复合句 | P2 | 需先完成简单句 | 待定 |
```

## 6. 合并流程（将新知识合并到已有节点）

当新源材料中的知识点与已有页面内容相关时，**禁止**创建重复页面，必须使用合并流程。

### 6.1 合并决策树

```
新知识点 vs 已有页面
│
├─ 完全相同？
│  └─ 是 → 添加来源引用，无需修改内容
│
├─ 互补（不重叠）？
│  └─ 是 → 在已有页面中添加新章节/表格/例句
│
├─ 部分重叠但有差异？
│  └─ 是 → 进入【冲突解决】（见 6.2）
│
└─ 同一概念的不同视角/教材描述？
   └─ 是 → 添加"多来源视角"小节，保留各来源原始表述
```

### 6.2 冲突解决规则（多来源矛盾）

当不同来源对同一知识点的描述矛盾时：

**规则 1: 标记冲突，不擅自裁决**
```markdown
> **来源差异注意**: 
> - Gaunt 教材描述：...
> - 清格尔泰语法描述：...
> - **待确认**: 两者在 ... 上存在差异。（status: needs-check）
```

**规则 2: 优先级裁决**
| 冲突类型 | 优先级 | 处理方式 |
|----------|--------|----------|
| 权威教材 vs 在线资源 | 教材优先 | 以教材为准，标注在线资源为"补充视角" |
| 现代用法 vs 传统语法 | 现代优先 | 以现代用法为准，传统用法标注为"书面/古典" |
| 口语 vs 书面 | 标注差异 | 两者并列，标注适用场景 |
| 同一作者不同版本 | 新版优先 | 以新版为准，旧版标注为"历史版本" |

**规则 3: 必须更新 `source` 字段**
```yaml
source: [[Gaunt 现代蒙古语]], [[蒙古语语法（西里尔）— 清格尔泰|清格尔泰语法]]
status: needs-check  # 当存在未解决冲突时
```

### 6.3 合并操作模板

在 log.md 中记录合并时：
```markdown
## [YYYY-MM-DD HH:MM] merge | 源材料名称 → 目标页面

- **源材料**: [[源材料页面]]
- **目标页面**: [[已有页面]]
- **合并类型**: 互补 | 冲突标记 | 视角补充
- **变更内容**:
  - 添加了: ...（具体章节/表格/例句）
  - 修改了: ...（具体段落）
  - 冲突标记: ...（如有）
- **状态变更**: needs-check → reviewed（或保持 needs-check）
```

## 7. 新建页面流程

当确定需要创建新页面时，遵循以下流程：

### 7.1 新建检查清单

```
□ 1. 确认页面类型（concept/entity/rule/comparison/example/synthesis/exercise）
□ 2. 确认页面名称不重复（搜索 wiki/ 中是否已有同名/同义页面）
□ 3. 编写完整 YAML frontmatter
□ 4. 按页面类型模板编写内容
□ 5. 添加至少 3 个 related 双向链接
□ 6. 例句遵循 4.1 强制格式
□ 7. 来源标注精确（教材名 + 章节/页码）
□ 8. 更新 wiki/index.md（添加到对应分类）
□ 9. 更新源摘要页的"已提取知识点"
□ 10. 记录到 wiki/log.md
```

### 7.2 页面类型内容模板

**Rule 页面**：
```markdown
# 标题 (术语 / 蒙古语)

## 规则概述
（一句话概括 + 适用场景）

## 规则详情
（表格、分类说明）

## 用法详解
（分场景，带例句）

## 例句
（遵循 4.1 格式）

## 常见错误
（列表形式）

## 关联知识
（双向链接）
```

**Entity 页面**：
```markdown
# 标题

## 概述
（定义 + 分类）

## 分类/列表
（表格或列表）

## 变化规则
（如适用）

## 例句
（遵循 4.1 格式）

## 关联知识
```

**Example 页面**：
```markdown
# 主题

## 核心句型
（3-5 个最常用句型，完整格式）

## 场景分类
（如"家庭成员""描述人物"）

### 场景A
（例句列表，标准格式）

### 场景B
（例句列表）

## 语法要点总结
（双向链接到 rule/concept 页面）
```

## 8. 双向链接规范

### 8.1 链接义务

- **Rule 页面** → 必须链接到相关的 Entity 和 Concept 页面
- **Example 页面** → 每个例句必须标注语法点（链接到 Rule 页面）
- **Source 页面** → 必须链接到已提取知识点的页面
- **Synthesis 页面** → 必须链接到所有被合成的源页面
- **Comparison 页面** → 必须链接到被对比的所有页面

### 8.2 链接维护

- 删除页面时，必须搜索并修复所有指向它的链接
- 重命名页面时，必须同步更新所有 wikilink
- 定期检查孤立页面（无入链的页面，index/log 除外）

## 9. 更新日志规范

`wiki/log.md` 是知识库的**审计追踪**，每次操作必须记录。

### 9.1 操作类型

| 类型 | 缩写 | 说明 |
|------|------|------|
| `ingest` | 收录 | 从源材料提取知识到知识库 |
| `merge` | 合并 | 将新知识合并到已有页面 |
| `create` | 创建 | 新建独立页面（非源自特定源材料） |
| `update` | 更新 | 对已有页面的增强或修正 |
| `lint` | 检查 | 运行健康检查并修复问题 |
| `enrich` | 丰富 | 跨页面补充内容（如添加对比页、合成页） |
| `fix` | 修复 | 修复错误（链接、混字、格式等） |
| `refactor` | 重构 | 大规模结构调整 |

### 9.2 日志条目格式

```markdown
## [YYYY-MM-DD HH:MM] <类型> | <简短描述>

- **来源/原因**: <源材料名 或 修改原因>
- **变更页面**:
  - 创建了: [[新页面1]], [[新页面2]]
  - 更新了: [[已有页面1]]（具体修改说明）
  - 合并了: [[源知识]] → [[目标页面]]
  - 删除了: [[废弃页面]]
- **建立链接**: [[页面A]] → [[页面B]]
- **健康检查**: 断裂链接 X, 缺 source/status Y, 混字 Z（或"通过"）
- **待办**: <如有未完成的后续工作>
```

### 9.3 日志纪律

- **每次操作必须记录**，无例外
- **变更必须具体**，不能写"更新了若干页面"
- **健康检查必须包含**，即使结果为"通过"
- **待办必须明确**，避免悬而未决

## 10. 边界与禁止事项

### 10.1 绝对禁止

| # | 禁止行为 | 原因 |
|---|---------|------|
| 1 | **修改 `raw/` 中的任何文件** | 原始材料是只读的，是审计追踪的根基 |
| 2 | **创建重复页面** | 同一知识点必须在单一页面维护 |
| 3 | **不记录 log.md 就提交变更** | 审计追踪断裂 |
| 4 | **不更新 index.md 就创建页面** | 导航失效 |
| 5 | **使用未验证的 OCR 文本作为来源** | 数据污染（如脚注数字合并问题） |
| 6 | **在例句中省略来源标注** | 无法追溯，学习者无法查证 |
| 7 | **擅自裁决来源冲突** | 必须标记待确认 |
| 8 | **删除已有页面而不修复入链** | 造成断裂链接 |

### 10.2 边界声明

- **不处理**: 老蒙文（传统蒙古文）的详细教学（除非用户明确要求）
- **不处理**: 蒙古语方言差异（默认以喀尔喀蒙古语/乌兰巴托口音为标准）
- **不处理**: 超出 `wiki/` 和 `schema/` 的文件（`raw/` 只读，`prd/` 由用户维护）
- **不自动**: 不自动将内容标记为 `reviewed`，必须由用户或明确的人工校验确认

## 11. 质量检查清单（Lint）

每次批量操作后，必须运行健康检查：

```
□ 断裂链接: [[链接]] 指向不存在的页面 → 修复或删除
□ 孤立页面: 无入链的页面（index.md / log.md 除外）→ 添加引用
□ 缺失元数据: concept/entity/rule 等缺少 source/status → 补齐
□ 格式问题: 缺少 frontmatter、type 错误 → 修复
□ 内容矛盾: 同一知识点在不同页面描述不一致 → 标记冲突
□ 混字污染: 西里尔单词中混入拉丁字母（如 -сан4）→ 修复
□ 过期内容: 标记为 draft 或 TODO 的内容 → 列出待完成
```

## 12. 快速参考

### 12.1 决策速查

```
用户说"收录 [文件]"
  → 检查 sources/ 是否存在
  → 检查 OCR 质量
  → 对每个知识点：已有页面？→ 合并 : 新建
  → 更新 index.md + log.md

用户说"更新 [页面]"
  → 读取现有页面
  → 判断是补充、修正还是重写
  → 更新 source 字段（添加新来源）
  → 更新 status 字段
  → 记录 log.md

用户说"检查知识库"
  → 运行 lint 脚本
  → 生成报告
  → 按优先级修复

发现来源冲突
  → 标记冲突（多来源视角小节）
  → 不擅自裁决
  → status → needs-check
```

### 12.2 紧急修复通道

以下问题属于 P0（立即修复）：
- 断裂链接 > 5 个
- 核心语法页面（如元音和谐律、名词变格）有事实错误
- 混字污染批量出现
- 页面缺失 frontmatter
