# 知识库扩展规范

> 本页面规定知识库在**新建知识节点**和**与已有文档合并**时的严格规则。
> 所有 AI 助手在扩展知识库时必须遵守。

---

## 1. 核心原则：单一事实来源（Single Source of Truth）

知识库的每个知识点**有且仅有**一个权威页面。当新来源提供同一知识点的信息时，禁止创建重复页面，必须合并到已有节点。

```
错误做法:
  来源A → 过去时-清格尔泰（单独页面）
  来源B → 过去时-Gaunt（单独页面）

正确做法:
  来源A → [[过去时]] ← 来源B（同一页面）
```

## 2. 源材料分级与准入

### 2.1 源材料三级体系

| 级别 | 名称 | 标准 | 处理策略 |
|------|------|------|----------|
| P0 | Primary | 权威教材、语法专著、学术文献 | 优先收录，作为"主要来源" |
| P1 | Secondary | 辅助教材、在线课程、权威词典 | 补充收录，作为"参考来源" |
| P2 | Tertiary | 博客、论坛、个人笔记、社交媒体 | 谨慎收录，**必须**交叉验证 |

### 2.3 新来源准入检查清单

任何新来源进入知识库前，必须完成：

```
□ 1. 创建/更新 Source 页面
   - 在 wiki/sources/ 中创建对应页面
   - 填写完整 YAML frontmatter
   - 标注级别（Primary/Secondary/Tertiary）

□ 2. 评估 OCR/文本质量
   - 文本可提取？→ 继续
   - 需要 OCR？→ 标记 status: needs-ocr，暂停收录

□ 3. 制定提取计划
   - 列出所有待提取知识点
   - 标注优先级（P0/P1/P2）
   - 标注依赖关系（如"需先完成简单句"）

□ 4. 用户确认（如需）
   - 新来源为 P2 级别时，建议用户确认
   - 提取计划复杂时，建议用户确认优先级
```

## 3. 新建节点流程（Create Flow）

当确认需要创建**全新**知识页面时，遵循以下严格流程。

### 3.1 前置检查（必须全部通过）

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 名称唯一性 | 搜索 wiki/ 所有文件，确认无同名/同义页面 |
| 2 | 类型正确性 | 按 CLAUDE.md §3.2 矩阵确认 type |
| 3 | 来源可追溯 | 能明确指向至少一个 Source 页面 |
| 4 | 内容独立性 | 该知识点不能作为已有页面的子章节 |

### 3.2 命名冲突解决

当新知识点与已有页面名称相似但不同时：

```
已有: [[过去时]] — 动词过去时（-сан/-ав/-аагүй）
新建: [[现在时构成]] — 现在/将来时（-на）→ 不同概念，可以新建

已有: [[名词变格]] — 8个格的总览
新建: "名词变格-给格" → ❌ 错误，应作为 [[名词变格]] 的子章节或归入 [[给格]]
```

**规则**: 如果新页面内容可以合理地作为已有页面的一个章节，则不新建页面。

### 3.3 新建页面最低内容标准

| 页面类型 | 最低要求 |
|----------|----------|
| concept | 定义（中文+蒙古语）+ 至少 1 个例子 + 关联知识（≥3个related链接） |
| entity | 分类/列表（表格）+ 至少 1 个例子 + 关联知识 |
| rule | 规则概述 + 规则详情（表格）+ 至少 2 个例句（完整格式）+ 常见错误 |
| comparison | 对比维度说明 + 形式对比表 + 最小对比例句 + 常见错误 |
| example | 核心句型（3-5个）+ 至少 5 个按场景分类的例句 + 语法要点总结 |
| synthesis | 覆盖范围说明 + 决策树/速查表 + 来源页面列表（≥3个）+ 适用阶段 |
| exercise | 至少 5 道题 + 答案 + 解析 + 关联知识点 |

### 3.4 新建后必须完成的操作

```
□ 1. 编写完整 YAML frontmatter
□ 2. 按页面类型模板编写内容
□ 3. 添加至少 3 个 related 双向链接
□ 4. 所有例句遵循完整格式（来源 + 语法点标注）
□ 5. 更新 wiki/index.md（添加到对应分类）
□ 6. 更新对应 Source 页面的"已提取知识点"
□ 7. 记录到 wiki/log.md
□ 8. 运行 lint 检查
```

## 4. 合并节点流程（Merge Flow）

当新来源中的知识点与已有页面相关时，**必须使用合并流程**。

### 4.1 合并类型判断

```
新内容 vs 已有页面
│
├─ Type A: 完全重复
│  └─ 已有页面已覆盖该内容
│     → 仅添加来源引用，无需修改内容
│     → 更新 source: 字段
│
├─ Type B: 互补不重叠
│  └─ 新内容填补已有页面的空白
│     → 在已有页面中添加新章节/小节
│     → 更新 source: 字段
│     → 例: 已有"离格-空间起点"，新来源补充"离格-比较用法"
│
├─ Type C: 同一概念的不同描述
│  └─ 不同教材用不同方式解释同一规则
│     → 添加"多来源视角"或"不同教材的表述"小节
│     → 保留各来源原始表述，不擅自统一
│     → 例: Gaunt 说"-сан 是口语最常用"，清格尔泰说"-сан 是陈述式过去时"
│
└─ Type D: 直接矛盾
   └─ 不同来源对同一规则给出矛盾的描述
      → 进入【冲突解决】（见 §5）
```

### 4.2 合并操作规范

**合并时必须记录的具体信息**：

```markdown
## [YYYY-MM-DD HH:MM] merge | <源材料> → <目标页面>

- **源材料**: Source页面（章节/页码）
- **目标页面**: 已有页面
- **合并类型**: A-重复补充 / B-互补增强 / C-视角补充 / D-冲突标记
- **具体变更**:
  - 添加了第X节：...
  - 修改了第Y节：...
  - 新增了表格：...
  - 新增了例句：...（数量）
- **来源字段更新**: 从 [旧来源] 更新为 [旧来源, 新来源]
- **状态变更**: 如 needs-check → reviewed（或保持 needs-check）
- **待确认项**: 如有冲突未解决，列出具体问题
```

### 4.3 合并后质量检查

合并完成后必须验证：

```
□ 已有页面的结构是否仍然清晰？
□ 新内容是否与旧内容自然衔接？
□ 所有例句是否都标注了正确的来源？
□ 是否存在未标注来源的新增内容？
□ 页面的 level 是否需要调整？
□ related 链接是否需要更新？
```

## 5. 冲突解决规则

当不同来源对同一知识点给出**矛盾**描述时，按以下规则处理。

### 5.1 冲突标记语法

在页面中添加专门的"来源差异"或"注意"区块：

```markdown
**来源差异注意**:

- **Gaunt 教材** (Ch.3 p.45): "-ав 在口语中极少使用"
- **清格尔泰语法** (§2.1.1): "-в 是陈述式过去时的书面形式"
- **Peace Corps** (Unit 2 p.37): "-сан 是回答 -ав 问句时的标准用法"

**待确认**: 各教材对 -ав/-в 的口语使用频率描述存在差异。
— status: reviewed
```

### 5.2 裁决优先级

| 冲突场景 | 优先级 | 处理方式 |
|----------|--------|----------|
| 权威语法专著 vs 通用教材 | **专著优先** | 以清格尔泰语法为准，教材标注为"教学简化" |
| 现代教材 vs 传统语法 | **现代优先** | 以现代用法为准，传统用法标注"书面/古典" |
| 口语教材 vs 书面语法 | **标注差异** | 两者并列，明确标注"口语中...""书面语中..." |
| 同一来源的不同版本 | **新版优先** | 以新版为准，旧版标注"历史版本" |
| 无法裁决的学术争议 | **全部保留** | 标记为"学术界存在争议"，保持 needs-check |

### 5.3 禁止行为

| # | 禁止 | 正确做法 |
|---|------|----------|
| 1 | 擅自选择一方删除另一方 | 保留双方，标注差异 |
| 2 | 将矛盾内容混在一起写成"和稀泥"表述 | 分条列出各来源原始表述 |
| 3 | 不标记冲突直接统一为一种说法 | 添加"来源差异注意"区块 |
| 4 | 将 P2 来源的冲突内容直接采纳 | P2 来源冲突必须经用户确认 |

## 6. 页面生命周期管理

### 6.1 状态流转

```
draft（草稿）
  → 内容基本完成 → needs-check（需校对）
    → 人工校对通过 → reviewed（已审阅）
    → 发现新来源冲突 → needs-check（退回）
  → 源材料需 OCR → needs-ocr（阻塞）
    → OCR 完成 → draft 或 needs-check
```

### 6.2 页面合并（合并两个已有页面）

当发现两个页面内容高度重叠时：

1. 选择一个作为主页面（通常选名称更通用、内容更完整的）
2. 将副页面的内容合并到主页面
3. 在副页面位置创建**重定向页**：
   ```markdown
   ---
   title: 旧页面名
   type: redirect
   ---
   
   本页面已合并到 主页面。
   ```
4. 更新所有指向旧页面的 wikilink
5. 记录到 log.md

### 6.3 页面删除

删除页面时必须：

```
□ 1. 搜索所有指向该页面的 wikilink
□ 2. 修复或删除所有入链
□ 3. 在 log.md 中记录删除原因
□ 4. 从 index.md 中移除链接
□ 5. 从相关 Source 页面的"已提取知识点"中移除
```

## 7. 快速决策表

| 场景 | 操作 | 参考章节 |
|------|------|----------|
| 新来源，无相关页面 | 新建页面 | §3 |
| 新来源，有互补内容 | 合并到已有页面 | §4.1 Type B |
| 新来源，有矛盾描述 | 标记冲突，不裁决 | §5 |
| 新来源，完全重复 | 仅添加来源引用 | §4.1 Type A |
| 发现两个重复页面 | 合并页面，创建重定向 | §6.2 |
| P2 来源与 P0 来源冲突 | P0 优先，P2 标注"待验证" | §5.2 |
| 页面内容已过时 | 更新并标注版本 | §6.1 |
