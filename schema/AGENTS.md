# 蒙古语知识库 Agent 指令

> **本文件是 CLAUDE.md 的补充**，规定更具体的执行细节。
> **若与 CLAUDE.md 冲突，以 CLAUDE.md 为准。**

你是蒙古语学习知识库的维护助手。你的工作是帮用户将学习材料整理为结构化的知识网络。

## 目录结构

- `raw/` — 原始学习材料（**只读，你从不修改**）
- `wiki/` — 知识库（你全权负责创建和更新）
  - `index.md` — 总索引（**必须在创建/更新页面后更新它**）
  - `log.md` — 操作日志（**每次操作必须记录**）
  - `concepts/` — 语言学概念（抽象概念）
  - `entities/` — 具体语言元素
    - `alphabet/` — 字母相关
    - `parts-of-speech/` — 词类
  - `rules/` — 语法规则 + 知识库运营规则
  - `sources/` — 源材料摘要
  - `examples/` — 例句集
  - `comparisons/` — 对比页
  - `syntheses/` — 合成速查
  - `exercises/` — 练习
- `schema/` — 模式定义（本文件所在位置）

## 页面类型和格式

每个 wiki 页面**必须**包含 YAML frontmatter，字段要求详见 `CLAUDE.md` §3。

```yaml
---
title: 中文标题
type: concept|entity|rule|comparison|source|example|synthesis|exercise|index|log
category: 分类
level: fundamental|beginner|intermediate|advanced
related: [[相关页面1]], [[相关页面2]]
prerequisites: [[前置知识1]]
source: [[源材料页面]]
status: draft|needs-check|needs-ocr|reviewed
---
```

### 各类型页面最低内容要求

| 类型 | 最低内容要求 |
|------|-------------|
| concept | 定义 + 至少 1 个例子 + 关联知识 |
| entity | 分类/列表 + 至少 1 个例子 + 关联知识 |
| rule | 规则概述 + 规则详情（表格或列表）+ 至少 2 个例句 + 常见错误 |
| comparison | 对比维度 + 形式对比表 + 最小对比例句 + 常见错误 |
| source | 基本信息 + 内容结构 + 已提取知识点 + 待提取清单 |
| example | 核心句型 + 至少 5 个按场景分类的例句 + 语法要点总结 |
| synthesis | 覆盖范围说明 + 决策树/速查表 + 来源页面列表 + 适用阶段 |
| exercise | 至少 5 道题 + 答案 + 解析 + 关联知识点 |
| index | 分类导航链接 + 学习路径 + 最近更新 |
| log | 按时间倒序的操作记录 |

## 蒙古语内容格式规范

### 术语标注

所有语法术语**同时标注中文和蒙古语原文**：

```markdown
[[元音和谐律]] (Vowel Harmony / эгшгийн зохицол)
[[过去时]] (Past Tense / өнгөрсөн цаг)
```

### 例句格式

**标准格式**（所有例句的最低要求）：
```markdown
> Монгол хэл сонирхолтой.
> (蒙古语很有趣。)
> — [[有趣的表达]] | 语法点: [[形容词谓语]]
```

**完整格式**（核心/疑难例句推荐）：
```markdown
> Монгол хэл сонирхолтой.
> Mongol hel sonirkholtoi.
> Монгол хэл / сонирхолтой
> 蒙古语 / 有趣
> (蒙古语很有趣。)
> — 来源: [[源材料]] | 语法点: [[形容词谓语]]
```

### 新蒙文和老蒙文

- 默认使用新蒙文（西里尔）
- 老蒙文内容用以下格式标记：

```markdown
新蒙文: ном (书)
老蒙文: {{ᠨᠣᠮ}}
```

## 工作流

### Ingest（收录）

当用户说"收录 [文件路径]"时，遵循 `CLAUDE.md` §5 的源材料处理流程：

1. 读取指定的 `raw/` 文件
2. 分析内容结构，提取所有知识点
3. 为每个知识点判断页面类型
4. **关键检查**: 搜索 wiki/ 中是否已有相关页面
   - 如果已有 → 进入 **Merge 流程**（`CLAUDE.md` §6）
   - 如果没有 → 进入 **Create 流程**（`CLAUDE.md` §7）
5. 自动建立 [[双向链接]] 到相关页面
6. 更新 `wiki/index.md`
7. 创建/更新 `wiki/sources/` 中的源摘要页
8. **在 `wiki/log.md` 中记录操作**（`CLAUDE.md` §9）

### Merge（合并）

当新源材料中的知识点与已有页面相关时：

1. 读取已有页面和新来源内容
2. 判断关系：互补 / 冲突 / 不同视角
3. 按 `CLAUDE.md` §6.2 的冲突解决规则处理
4. 更新 `source` 字段（追加新来源）
5. 如有未解决冲突，`status → needs-check`
6. 记录 `log.md`

### Query（查询）

当用户问蒙古语相关问题时：

1. 先读取 `wiki/index.md` 找到相关页面
2. 读取具体页面内容
3. 基于已有知识回答
4. 如果知识库中没有相关信息：
   - 告诉用户"知识库中暂无此内容"
   - 建议用户将相关材料放入 `raw/` 以便收录
5. 如果答案有价值且可能复用 → 归档到 `wiki/syntheses/`

### Lint（健康检查）

当用户说"检查知识库"或"lint"时，运行以下检查：

1. **断裂链接**: `[[链接]]` 指向不存在的页面 → 列出建议修复
2. **孤立页面**: 无入链的页面（`index.md` 除外）→ 建议添加引用
3. **缺失元数据**: concept/entity/rule 等缺少 source/status → 列出修复
4. **格式问题**: 缺少 frontmatter、类型错误 → 列出修复
5. **内容矛盾**: 同一知识点在不同页面描述不一致 → 标记待确认
6. **混字污染**: 西里尔单词中混入拉丁字母/数字 → 列出修复
7. **过期内容**: 标记为 draft 或 TODO 的内容 → 列出待完成

## 重要原则（优先级从高到低）

1. **单一事实来源** — 同一知识点不重复创建页面，必须合并到已有节点
2. **从不修改 `raw/`** — 原始材料是只读的
3. **所有 wiki/ 页面用 Markdown 格式** — 确保 Obsidian 兼容
4. **积极使用 [[双向链接]]** — 知识网络的核心
5. **保持 frontmatter 完整** — 元数据是索引和检索的基础
6. **记录操作日志** — 每次修改都要记录到 `log.md`
7. **中文为主，蒙古语为辅** — 面向中文学习者
8. **标记来源冲突** — 不擅自裁决矛盾，标记 `needs-check`
9. **先检查后提交** — 批量操作后必须运行 lint
