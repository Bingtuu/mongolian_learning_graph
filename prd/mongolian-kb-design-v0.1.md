## 1. 卡帕西 AI 知识库理念研究

### 1.1 核心理念：LLM Wiki

2025年4月，Andrej Karpathy（OpenAI 创始成员、前特斯拉 AI 负责人）发布了一份 GitHub Gist，提出了 **LLM Wiki** 模式——一种利用大语言模型构建和维护个人知识库的新范式。该 Gist 在48小时内获得5,000+ Stars，衍生出5,000+ Forks，形成了完整的开源社区生态。

**核心洞察**：

> 传统 RAG（检索增强生成）的问题在于：LLM 每次都需要从原始文档中重新发现知识，没有知识积累。LLM Wiki 的核心思想是让 LLM **增量式地构建和维护一个持久的维基**——一个结构化的、相互链接的 Markdown 文件集合，位于用户和原始源之间。

### 1.2 与传统 RAG 的对比

| 维度 | 传统 RAG | LLM Wiki |
|------|---------|----------|
| **知识处理方式** | 查询时实时检索原始文档片段 | 预先编译为结构化知识 |
| **知识积累** | 无积累，每次都从零检索 | 知识持续复合增长 |
| **交叉引用** | 查询时动态生成 | 预先建立并持续维护 |
| **一致性** | 依赖检索质量，结果不稳定 | 知识已加工、审核、组织 |
| **维护成本** | 需要复杂的向量数据库和索引 | Markdown 文件即可，无需向量库 |
| **查询响应** | 需要检索+生成两步 | 直接读取已凝练好的内容 |

### 1.3 三层架构

Karpathy 提出了极简却极具工程美感的三层解耦架构：

```
raw/          ← 原始源材料（不可变，LLM 从不修改）
    ↓         
wiki/         ← LLM 维护的知识库（Markdown，LLM 全权拥有）
    ↓         
schema/       ← 模式定义（AGENTS.md / CLAUDE.md，定义工作流）
```

**raw/ — 不可变层**: 存储原始学习材料——教材章节、词典条目、语法规则、例句、音频文本等。这些是真相的唯一来源。

**wiki/ — LLM 维护层**: 由 LLM 生成和维护的知识页面，包括：
- **索引文件** (index.md): 内容导向的目录，每页带链接和摘要
- **概念文章** (concepts/): 按主题组织的概念页面（如"元音和谐律"、"格变化"）
- **实体页面** (entities/): 具体条目（如特定词汇、语法规则、例句集）
- **源摘要** (sources/): 每份原始材料的摘要页面
- **合成页面** (syntheses/): 查询结果被归档为可复用的知识

**schema/ — 模式控制层**: 告诉 LLM 如何维护 wiki 的配置文件，包括：
- 页面类型和格式规范
- 收录 (ingest)、查询 (query)、检查 (lint) 工作流
- 交叉引用规则和维护约定

### 1.4 四大操作循环

| 操作 | 功能描述 | 在蒙古语知识库中的应用 |
|------|---------|---------------------|
| **Ingest (收录)** | LLM 读取 raw/ 新材料，提取关键信息，整合到 wiki | 新增词典条目、语法规则、例句集 |
| **Query (查询)** | 基于 wiki 内容回答问题，输出归档回 wiki | 学习者查询语法问题、词汇辨析 |
| **Lint (健康检查)** | 扫描不一致、断裂链接、孤立页面、内容缺口 | 检查词条矛盾、过期内容、关联缺失 |
| **Enrich (丰富)** | 主动发现新知识，通过网络搜索填补缺口 | 自动补充新词汇、发现新的语法现象 |

### 1.5 反馈循环——知识复利

```
新学习材料 → Ingest → 更新 Wiki 页面 → 交叉引用更新
                                              ↑
用户查询 ← Query ← 从已编译知识回答 ────────┘
  │
  └─→ 归档到 Wiki (syntheses/) → 知识积累
```

**关键洞察**: 每一次查询都让维基变得更好。知识不是消耗品，而是投资品。

### 1.6 为什么这适用于语言学习

Karpathy 的模式特别适合蒙古语学习知识库，原因如下：

1. **语言知识的结构性**: 语法规则、词汇关系、构词法天然具有网络结构，适合 wiki 形式
2. **知识间的强关联性**: 元音和谐律影响格变化、格变化影响动词变位——关联是理解的关键
3. **渐进式学习**: 与新蒙文优先、后扩展老蒙文的策略天然一致
4. **知识的复利效应**: 每新增一个例句都可能关联到多个语法点

## 3. 可行性分析

### 3.1 整体可行性：高

基于卡帕西 LLM Wiki 理念构建蒙古语学习知识库在技术、资源和时间维度上均具有**高度可行性**。

**核心依据**：
1. **模式验证充分**: LLM Wiki 已被社区广泛验证，有多个开源实现可参考
2. **蒙古语 NLP 基础设施成熟**: 有 BERT、ALBERT 等预训练模型和多个数据集
3. **学习资源丰富**: 大量免费/开源的词典、语法书、课程可作为 raw/ 层材料
4. **技术栈简洁**: 核心仅需 Markdown + LLM，无需复杂的 RAG 基础设施

### 3.2 新蒙文优先策略

| 维度 | 新蒙文（西里尔） | 老蒙文（传统竖排） |
|------|-----------------|-------------------|
| **技术复杂度** | 低 - 标准 Unicode，水平排版 | 高 - 需要特殊字体，竖排布局 |
| **Unicode 支持** | 完整，现代系统原生支持 | 需特殊字体和渲染支持 |
| **NLP 工具链** | 成熟 - 有 BERT、分词工具等 | 较薄弱，语料稀缺 |
| **开源资源** | 丰富 | 有限 |
| **学习者需求** | 初学者首选（蒙古国官方文字） | 进阶/文化学习者 |
| **实现阶段** | **第一阶段** | **第二阶段** |

### 3.3 技术可行性矩阵

| 功能模块 | 可行性 | 依赖条件 | 风险等级 |
|---------|--------|---------|---------|
| Markdown 知识库核心 | ★★★★★ | 无 | 极低 |
| LLM 自动收录 (Ingest) | ★★★★★ | LLM API 访问 | 低 |
| 交叉引用自动维护 | ★★★★☆ | 良好的 schema 设计 | 低 |
| Web 前端浏览/搜索 | ★★★★★ | 参考开源实现 | 极低 |
| 知识图谱可视化 | ★★★★☆ | Obsidian 或 D3.js | 中 |
| 新蒙文内容处理 | ★★★★★ | 标准 Unicode 支持 | 极低 |
| 老蒙文渲染 | ★★★☆☆ | Noto 字体 + 特殊布局 | 中 |
| 语音合成 (TTS) | ★★★☆☆ | MnTTS 数据集 | 中 |
| 拼写检查 | ★★★★☆ | mongolian-spellcheck | 低 |
| 西里尔↔传统转换 | ★★★★☆ | bichig2cyrillic 模型 | 中 |

### 3.4 潜在风险与缓解

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| LLM 对蒙古语理解不准确 | 知识库质量下降 | 人工审核 + 蒙古语 NLP 模型校验 |
| 开源资源授权混杂 | 法律风险 | 严格筛选开源许可证，分离非开源引用 |
| 传统蒙古文渲染不一致 | 用户体验差 | 优先实现新蒙文，老蒙文使用成熟字体 |
| 知识库膨胀导致查询变慢 | 性能问题 | 按规模演进（index → qmd 混合搜索 → 向量库） |
| 社区贡献不足 | 内容增长慢 | 种子内容由 LLM 从开源资源自动生成 |

## 4. 设计理念与核心原则

### 4.1 知识组织范式：从线性到网络

传统语言学习是线性的：教材 → 章节 → 练习。AI 知识库则是**网络化的**：

```
                    [蒙古语概述]
                   /      |      \
            [语音]    [文字]    [语法]
             / \       / \       / \
        [元音] [辅音] [新蒙] [老蒙] [名词]
         /             |             \
    [元音和谐]      [西里尔字母]    [格变化]
       /                               \
  [前后元音]                          [8个格]
       \                               /
        \--- [关联: 元音和谐影响格后缀] ---/
```

**每个知识节点都明确知道自己在网络中的位置，以及与其他节点的关联。**

### 4.2 核心设计原则

#### 原则 1：知识即图 (Knowledge as Graph)

知识不是孤立的条目，而是相互连接的图。每条语法规则都链接到相关例句、词汇变化和对比规则。图的可视化让学习者看到"知识的全貌"。

#### 原则 2：LLM 维护，人类策展 (LLM Maintains, Human Curates)

- **人类的职责**: 选择优质源材料、提出关键问题、判断内容重要性、审核准确性
- **LLM 的职责**: 总结、交叉引用、归档、健康检查——所有繁琐的维护工作

#### 原则 3：文件优先于应用 (File over App)

知识库本质上是 Markdown 文件 + Git。不锁定在任何特定工具中，可被任何编辑器、搜索引擎、LLM 读取。

#### 原则 4:渐进式复杂度 (Progressive Complexity)

从最简单可行的方案开始，按规模演进：

| 规模 | 策略 | 搜索方式 |
|------|------|---------|
| < 100 源文档 | 纯 Markdown + LLM 上下文 | index.md 索引 |
| 100-500 文档 | qmd 混合搜索 (BM25 + 向量) | CLI 搜索工具 |
| 500+ 文档 | 引入轻量向量库 | 全文搜索引擎 |

#### 原则 5:新蒙文优先，老蒙文兼容 (Cyrillic First, Script Compatible)

第一阶段聚焦新蒙文（技术门槛低、资源多、学习者需求大），架构设计上预留老蒙文的扩展空间。

### 4.3 页面类型设计

基于 Karpathy 的模式，蒙古语知识库采用以下页面类型：

| 页面类型 | 用途 | 示例 |
|---------|------|------|
| **概念页 (concept)** | 解释抽象语言学概念 | `[[元音和谐律]]`、`[[黏着语]]` |
| **实体页 (entity)** | 具体的语言元素 | `[[西里尔字母 А]]`、`[[名词]]` |
| **源摘要 (source)** | 每份原始材料的摘要 | `[[Tserenpil 语法书第3章]]` |
| **规则页 (rule)** | 语法规则条目 | `[[名词的复数形式]]` |
| **例句集 (examples)** | 按主题组织的例句 | `[[日常问候例句]]` |
| **对比页 (comparison)** | 易混淆知识对比 | `[[ө vs о 发音对比]]` |
| **合成页 (synthesis)** | 查询结果归档 | `[[格变化总览]]` |

### 4.4 关联类型设计

```
[元音和谐律] ── prerequisite ──> [名词变格]

# 名词的复数形式

## 规则概述

蒙古语名词复数通过添加后缀构成，后缀选择取决于词尾元音。

## 复数后缀表

| 词尾元音类型 | 后缀 | 示例 |
|-------------|------|------|
| 后元音 (a, o, u) | -уд / -үүд | хүн → хүмүүс |
| 前元音 (e, ö, ü) | -үүд / -үүд | морь → морьд |
| 特殊 | -нар / -нер | багш → багш нар |

## 注意事项

- 以辅音结尾的名词加 -үүд
- 元音和谐律决定后缀形式
- 参见 [[元音和谐律]] 了解元音分类

## 例句

> (很多人在蒙古生活。)
> -- [[日常问候例句]]

### 5.5 渐进式实现哲学

整个知识库从一个**当天就能用起来**的本地工具开始，逐步演化为面向用户的产品。每一阶段都是完整可用的系统，不依赖下一阶段。核心原则：

- **先用手动，再用自动**：先用人工方式跑通流程，再用 LLM 自动化
- **先用文件，再用数据库**：先用 Markdown 文件，数据量大了再引入数据库
- **先用本地，再上线**：先在本地使用和验证，确认价值后再部署上线
- **先用新蒙文，再扩展老蒙文**：优先实现新蒙文，架构上预留老蒙文扩展

# 元音和谐律 (Vowel Harmony)

## 核心概念

蒙古语是[[黏着语]]，通过[[后缀]]表达语法意义。元音和谐律决定了后缀中元音的选择。

## 元音分类

蒙古语元音分为两大类：

| 后元音 (Back) | 前元音 (Front) | 中性 |
|--------------|---------------|------|
| а, о, у      | э, ө, ү       | и    |

## 规则

词根中的元音决定了后续[[后缀]]的元音类型：
- 含后元音的词 → 接后元音后缀：хүн**үүд** (人们)
- 含前元音的词 → 接前元音后缀：ном**үүд** (书们)
- 含中性元音 + 后元音 → 后元音后缀
- 含中性元音 + 前元音 → 前元音后缀

## 关联知识

- [[名词的复数形式]] — 复数后缀受元音和谐影响
- [[名词变格]] — 格后缀受元音和谐影响
- [[后缀]] — 蒙古语语法核心机制

## 例句

> Монгол хэлэнд [[元音和谐律|元音和谐]] маш чухал.
> (在蒙古语中，元音和谐非常重要。)
```

**方案 C：完整 SaaS**
- 独立部署的后端服务
- 数据库（PostgreSQL）
- 向量搜索（pgvector）
- 用户认证系统

#### 阶段 3 交付物

- [ ] 交互式蒙古语学习网站
- [ ] AI 问答助手（蒙古语语法专家）
- [ ] 知识图谱探索界面
- [ ] 学习路径推荐
- [ ] 开源发布或 SaaS 上线

### 四阶段演进总览

| 维度 | 阶段 0: 本地库 | 阶段 1: AI增强 | 阶段 2: 静态网站 | 阶段 3: 产品 |
|------|--------------|--------------|----------------|-------------|
| **时间** | 第 1 天 | 1-2 周 | 2-4 周 | 2-3 个月 |
| **技术复杂度** | 极低 | 低 | 中 | 中高 |
| **成本** | 免费 | LLM API 费用 | 免费托管 | 服务器费用 |
| **用户** | 你自己 | 你自己 | 任何人（只读）| 注册用户 |
| **AI 自动化** | 无 | Ingest/Query/Lint | 构建时生成 | 实时交互 |
| **维护者** | 你 | 你 + AI | AI + CI | 系统 + 社区 |
| **核心工具** | Obsidian | Obsidian + Claude | Quartz/VitePress | React + 后端 |
| **退出信号** | 手动太繁琐 | 想分享给别人 | 需要动态功能 | 持续迭代 |

### 关键决策点

在每个阶段结束时，你需要做一个关键决策：

**阶段 0 → 1**: "我是否愿意引入 LLM 来减少维护工作？"（需要 API Key 和少量费用）

**阶段 1 → 2**: "我是否希望其他人也能访问这个知识库？"（需要学习静态站点工具）

**阶段 2 → 3**: "是否有足够的需求 warrant 一个完整产品？"（需要评估用户反馈和投入产出比）

**核心原则**: 你在每个阶段都可以停下来长期使用。这不是一个"全有或全无"的项目，而是一个可以持续使用、逐步增强的工具。

### 7.4 设计哲学参考

> "The tedious part of maintaining a knowledge base is not the reading or the thinking — it's the bookkeeping. Updating cross-references, keeping summaries current, noting when new data contradicts old claims, maintaining consistency across dozens of pages. Humans abandon wikis because the maintenance burden grows faster than the value. LLMs don't get bored, don't forget to update a cross-reference, and can touch 15 files in one pass. The wiki stays maintained because the cost of maintenance is near zero."
> — Andrej Karpathy

> "知识库的难点从来不是阅读与思考，而是繁琐的维护工作。而维护，恰恰是 AI 最擅长的领域。"
